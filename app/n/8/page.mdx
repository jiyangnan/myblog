## 2. 第一次连接（非常关键）

### ✅ 正确操作顺序

1. 打开 Chrome -> 进入 `https://x.com/home`
2. 点击浏览器工具栏上的 **OpenClaw Relay** 图标
3. 确认图标徽标显示 **ON**（表示已连接）

**判定标准**：

- 没亮 -> 系统无法发推
- 亮了 -> 才允许自动流程继续

## 3. 大坑复盘：浏览器插件 vs CDP

我们真实走过一圈“换方案 -> 折腾 -> 回退”。

### 方案 A：浏览器插件（Relay）

**优点**：

- 最稳定、最贴近真实用户操作
- 不需要额外启动参数

**缺点**：

- 需要人工点亮 Relay
- 连接偶尔会断

### 方案 B：CDP（远程调试端口）

**尝试原因**：

- 想无人值守，不依赖人工点 Relay

**踩坑点**：

- 需要 `--remote-debugging-port=9222` 启动 Chrome
- 容易触发登录异常/风控
- 稳定性不如 Relay，反复“登录要求”

**最终结论**：

👉 **回归插件模式**（稳定优先，人工点亮可接受）

**方案对比图：**

![image.png](https://raw.githubusercontent.com/jiyangnan/picBed/main/img/20260212183959984.png)

> 经验：自动化里最重要的是“可持续交付”，不是“完全无人值守”。

## 5. 正常发推流程长什么样？

每次发推会自动跑以下流程：

1. **抓时间线** -> 取最近 10-20 条
2. **写推文** -> 按“先说看到什么，再给可执行方案”
3. **生成白板图** -> Nano Banana 出图
4. **自动发布** -> 发布后自动校验

## 6. 常见错误与解决办法（最实用）

### ❌ 情况 A：发推卡住，提示 gateway 不可达

**解决方法：**

1. OpenClaw 菜单栏 -> Gateway -> **Stop -> Start**
2. 回到 Chrome -> 确保 Relay 仍 ON
3. 再次重试

### ❌ 情况 B：提示 Relay 未连接

**解决方法：**

- 打开 x.com -> 点 Relay -> 徽标 ON

### ❌ 情况 C：白板图出现草地/天空

**原因**：脚本里曾遗留默认场景提示词

**解决方法**：已修复，并加“严格遵循提示词”强约束

### ❌ 情况 D：推文太长被截断

**解决方法：**

- 要么缩短
- 要么发线程（可自动分线程）
